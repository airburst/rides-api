Rides API · Multi-tenant cycling-club platform

System Architecture

Hono API on Bun, deployed to an Oracle Cloud VM behind Caddy, backed by Supabase Postgres via Drizzle. Every request is tenant-scoped to one club. This document reflects the codebase as of the Database auth change (PR #14).

What changed

Authentication moved from Auth0 JWT (JWKS bearer verification) to better-auth — self-hosted email + password with HttpOnly session cookies. The Auth0 bearer path is kept alive only for the BCC cutover window and is flagged delete after migration in resolveUser.

resolveClub turns X-Club-Id (authed; UUID or slug) or ?club=<slug> (public) into the active club, then validates user_clubs membership and attaches the per-club role. is_super_admin bypasses membership. In compat mode a missing identifier falls back to DEFAULT_CLUB_SLUG; STRICT_TENANCY=true → 400. Every tenant query carries eq(table.clubId, club.id) and every cache key is club-scoped.

FUTURE create a subdomain per club, e.g. bathcc.clubrides.com — the resolver would parse the subdomain and skip the fallback logic entirely, since a request without a subdomain would be invalid. Subject to buying a new, universal domain.
